## 0. Orientation

The Brain is a repository and tooling layer that keeps work coherent across
multiple agent sessions. It has no inherent mission — the operator defines
identity in `brain/persona/persona.md` and current goals in
`brain/goals/goals.md`. This manual describes how the framework operates
regardless of what the operator is building.

Every agent seat (Claude Code, Codex, Gemini) operates on the same repository
under the same rules. Only the invocation mechanics differ: slash commands
vs. CLI-direct vs. bootstrap scripts. The rules in Sections 1 through 6 and
8 through 14 are the same for every seat. Section 7 enumerates the
differences that cannot be unified.

**What this manual is not.** It is not the memory-policy source of truth
(that's `brain/wiki/pages/memory-policy.md`). It is not the wrap-up tool page
(that's `brain/wiki/tools/brain-meditate.md`). It is not the governance ADR
log (`brain/governance/`). It cites those files where they are authoritative.
When this manual and another file disagree, fix the disagreement; do not
silently ignore it.

### 0.1 Communication style

Every seat, in every response, in every surface:

- Be concise and direct.
- **No em dashes.** Use `--` instead.
- State what you did, not what you are about to do.
- Lead with the outcome when reporting results.
- **On success:** return `completed` plus one compact evidence line when
  useful.
- **On failure:** return the failing command, the key error line, and one
  remediation command.

{sources: A-012, A-013}

### 0.2 Multi-agent environment

The repository is shared by four agents. Know who does what before writing
about another seat:

- **Claude Code.** Primary builder. Hooks, skills, MCP tools, auto-memory.
  Currently the main laptop seat.
- **Codex.** Active executor seat. Runs from `${CODEX_LAB_PATH:-~/codex-lab}/` but owns the
  laptop seat when `brain-codex-enter` claims it. **Not** read-only.
- **Gemini.** Fourth seat, guest agent, independent rate limits. Routing
  tiers A/B/C (§7.3).
- **Clip.** Discord community AI running on the production server. Not a
  laptop seat.
- **ChatGPT.** External advisor. No repo access.

Seat state is in `brain/ops/.brain-active-seat.json` (§6.1). Exactly one
agent owns the laptop seat at a time.

## 2. Truth, State, Inbox

Three rules, non-negotiable.

### 2.1 Operational truth

Check live state first. Never make operational claims from memory alone. If
this manual says "`brain-resume` writes `session-resume.md`" and you suspect
the behavior has changed, re-read the script or run it. Memory is a hint,
not a source.

### 2.2 Verdict gate

Before mutating brain state, verify `brain/ops/derived/state-verdict.json`
is less than 5 minutes fresh.

| Verdict | Action |
|---|---|
| `OK` | Allow mutation. |
| `STALE` | Warn the operator, proceed with care, mention the verdict in the response. |
| `DRIFT` | **Block.** Investigate drift source. Do not mutate until resolved. |
| `CONFLICT` | **Block.** Investigate conflict source. Do not mutate until resolved. |

Refresh verdict via `brain-state-check`. `brain-meditate` stage 1 and
`brain-resume` surface the verdict automatically.

### 2.3 Inbox and triage

`work/inbox/` holds unprocessed items. **Read before archiving.** Promote
items to the task queue via `brain-task-admit` once admission metadata is
complete. Required fields: `created_by`, `creation_mode`, `authority_basis`,
`why_now`.

{sources: C-007, C-008, G-020, G-022, G-049, brain-task-admit invariants}

### 2.4 Capability reference resolution

When the operator refers to an existing tool or capability by description
("use the tool created", "use the thing we built", "use the export tool",
"use the tool from that task"), **do not guess or assume**. Resolve first
through the brain's own discovery surfaces, in this order:

1. `brain/wiki/tools/MAP.md` — scan for matching tool name or description
2. `brain/ops/registry/agent-tool-registry.json` — machine-readable registry, searchable by name/category
3. Completed task history (`brain/ops/tasks/completed/`) — grep titles for the capability described
4. `brain/wiki/tools/<tool>.md` pages — if a candidate name is found, confirm via its wiki page

**Decision rule:**

- If exactly one tool matches: use it. Do not ask.
- If multiple plausible tools remain after lookup: ask one narrow question naming the candidates.
- If the discovery surfaces produce nothing after a genuine search: ask.

**Never skip this lookup and jump to a broad assumption.** The brain maintains
these surfaces precisely so seated agents do not need to guess at operator
references. A speculative guess that bypasses discovery is a governance failure,
not a reasonable default.

## 4. Task Lifecycle

Tasks flow `queue/ -> active/ -> completed/`. Every AI-owned task follows
the same lifecycle. The lifecycle is tool-owned; **never edit `status:`
manually.**

### 4.1 Admission

Create and admit in one step:

```bash
brain-task-quick "Task title" --node <node> [--p1|--p2|--p3|--p4] \
  --why "reason" --steps "step 1;step 2;step 3"
```

Or hand-author the task file and admit:

```bash
brain-task-admit <path/to/task.md>
```

Admission-required fields: `created_by`, `creation_mode`, `authority_basis`,
`why_now`. Auto-generated tasks have all fields hard-required. Manual tasks
emit warnings on missing fields unless `--strict` is passed.

### 4.2 Claim

```bash
brain-task-claim <TASK-ID> <agent> [estimate_minutes]
# or, to claim and immediately start an execution wrapper:
brain-task-execute <TASK-ID> --by <agent>
```

Either call moves the file `queue/ -> active/` and sets `status=active` and
`owner=<agent>`. If a task is stuck in `active/` with a stale owner,
reclaim with `--force`.

### 4.3 Execute

Do the work described in the task. Admit follow-up tasks via
`brain-task-quick` as they emerge -- do not accumulate ad-hoc work in the
current task body.

### 4.4 Complete

```bash
brain-task-complete <TASK-ID> --by <agent> \
  --output "summary" --notes "implementation notes"
```

This tool owns the `active -> completed` transition and the
`status=active -> status=completed` flip.

**Post-Task Assimilation Audit (Mandatory for core work).** Before completing
any task involving architectural, governance, or core-infrastructure changes,
agents MUST perform the assimilation check defined in:
`brain/governance/post-task-assimilation-audit.md`.
Recommended for all other tasks.

**Immutability and Governed Repair (Option B).** Completed tasks are immutable
canonical records. On the first scan after completion, `brain-state-check`
registers a SHA256 checksum. Any subsequent modification triggers an
`IMMUTABILITY_VIOLATION`.

Controlled repair is allowed ONLY via the following protocol:
1. Identify the root cause of the violation.
2. Create an **Amendment Artifact** at:
   `work/logs/post-completion-amendments/<TASK_ID>.md`
3. Include: correction summary, reason for mutation, and original task link.
4. `brain-state-check` will automatically detect the amendment and authorize a
   checksum update (`IMMUTABILITY_REPAIR_APPLIED`).
5. Silent or undocumented mutations remain a hard drift block.

### 4.5 HARD CONTRACT: no manual status flipping

Do **not** pre-set `status: completed` in the task file.
`brain-task-complete` has a precondition that refuses to run if the task is
already completed and emits `Error: status=completed (must be active or
blocked)`. Recovery: edit `status: completed` back to `status: active` and
re-run `brain-task-complete`. This is intentional precondition defense
against the "manual completion drift trap".

### 4.6 Proof of execution (AI-owned tasks)

AI-owned tasks with an execution contract require a proof run:

```bash
RUN_ID=$(brain-proof-start --task <TASK-ID> --executor <agent>)
brain-proof-log --run "$RUN_ID" --event command --payload-json '{"cmd":"...","exit_code":0}'
brain-proof-log --run "$RUN_ID" --event file_write --payload-json '{"path":"..."}'
brain-proof-verify --run "$RUN_ID"
```

Verdict behavior: `FAIL` blocks completion, `WARN` requires an explicit
note, `PASS` is the normal path. Proof runs are **not** required for
orchestration-only work (e.g. handoff prompt generation): PoE applies at
task-execution boundaries, not at prompt preparation.

### 4.7 Remediation Authority Gating (Gating under Ambiguity)

When handling remediation or ambiguous backlog states, all agents must follow the
mandatory gating rules (Validate-then-Mutate) found in:
`brain/governance/remediation-authority-gating.md`.

**The Three Gates:**
1. **Creation Gate.** Do not create remediation tasks from a `SUSPECT_SOURCE`
   unless framed as validation-only.
2. **Mutation Gate.** No archive/release/reclassify based on spot-checks alone.
   Requires independent confirmation and an authorized cleanup lane.
3. **Ambiguity Default.** Default to recording state, classifying sources, and
   prohibiting mutation/execution.

**Taxonomy.** Explicitly label surfaces as `VALIDATED_SOURCE`, `SUSPECT_SOURCE`,
`MIXED_HOLDING_AREA`, or `CONFIRMED_ORPHAN_WITH_AUTHORIZED_CLEANUP`.

## 5. Handoff Contract

> **Scope:** optional multi-seat orchestration add-on. A single-seat brain
> can skip this entire section; the task lifecycle (§4) works standalone.
> Read on if you orchestrate multiple Claude/Codex/Gemini seats against a
> shared lane registry.

The handoff system is brain-native. Handoff state lives at
`$BRAIN_DIR/brain/ops/handoffs/` (default `~/brain/brain/ops/handoffs/`).
The system generates deterministic artifacts from a lane registry so
seats can execute pending lanes only, filter by priority, and seed tasks
without opening run state.

**Legacy bridge (compatibility-only):** scripts honor `HANDOFF_REGISTRY`,
`HANDOFF_ROOT`, and `CODEX_LAB_PATH` env overrides for reading from a
legacy `~/codex-lab/handoffs/` location. These are opt-in only; defaults
point at the brain-native store.

**Name disambiguation.** `/brain-handoff` as a slash command (and its Codex
emulation) invokes the handoff **system**. `brain-handoff` as a bin tool at
`~/bin/brain-handoff` is a separate legacy session-notes tool and is
**not** the orchestration entrypoint. When this manual says "handoff
system" or "`/brain-handoff`", it means the brain-native orchestration
path rooted at `brain/ops/handoffs/`.

### 5.1 Runtime paths

| Path | Purpose |
|---|---|
| `$BRAIN_DIR/brain/ops/handoffs/handoff-lanes.json` | Lane registry (canonical, brain-native). |
| `$BRAIN_DIR/brain/ops/handoffs/active-next-handoff.md` | **Execution source.** Generated execution ledger. |
| `$BRAIN_DIR/brain/ops/handoffs/active-next-handoff-prompt.txt` | Generated prompt. |
| `$BRAIN_DIR/brain/ops/handoffs/structured-handoff.md` | **Operator decision guide.** Distinct from the execution source above; use this for picking what to run under time or credit constraints, not for execution itself. |
| `~/bin/brain-handoff-run-all` | Run-all helper. |
| `~/bin/brain-handoff-seed` | Seed helper. |
| `~/bin/brain-handoff-validate` | Refresh + validate. |
| `~/bin/brain-handoff-parity-gate` | Operational parity gate (start/close). |
| `~/codex-lab/handoffs/*` | **Legacy, compatibility-only.** Read via `CODEX_LAB_PATH` / `HANDOFF_ROOT` env overrides. No live writes. |

### 5.2 Modes

| Mode | Behavior |
|---|---|
| `show` | Read-only. Build and present the prompt. |
| `run [--agent X] <lane-ids>` | Execute specified pending lanes. `--agent` defaults to `$BRAIN_AGENT` when omitted. |
| `run-all [--agent X] [--priority Pn\|--max-priority Pn]` | Execute pending lanes sequentially with per-lane closeout, optionally filtered. |
| `seed-tasks [--lane L] [--max N] [--priority Pn\|--max-priority Pn]` | Create queued tasks from pending lanes only. No execution. |

### 5.3 Freshness gate (before ANY mode)

1. Start parity gate:
   `bash ~/bin/brain-handoff-parity-gate start --quick`
   Read the emitted report at
   `codex-lab/system-reports/operational-parity-gate-start-<ts>.md`.
   **Stop immediately** if the report shows `Verdict: FAIL` or any
   `[FAIL]` subcheck line. The parity gate exits 0 regardless of
   verdict (and the subcheck tools `handoff-system-check.sh` and
   `memory-integrity-gate.sh` have the same pattern when called
   directly) -- do not trust exit code alone.
2. Refresh and validate:
   `bash ~/bin/brain-handoff-validate`
   Non-zero exit: **stop immediately**.
3. Read `handoff-lanes.json` fresh with `Read`. Never rely on prior
   in-session memory of lane state.
4. **Empty-pending fallback.** If `pending_count=0`, do not end the
   session. Report "no pending lanes", then resume any active Brain task
   owned by the current agent (reclaim with `--force` if stale), then
   continue into admitted queue tasks. If there are no runnable queue
   tasks either, stop cleanly with "no runnable work".
5. Non-empty: print pending lane IDs with priorities and continue.

**Stop condition summary (applies across all steps).** Stop if ANY step
exits non-zero OR emits a report with `Verdict: FAIL` / `[FAIL]`
subcheck. The umbrella `operational_parity_gate.sh` and its direct
subcheck tools exit 0 regardless of verdict, so the report file is the
source of truth. See `brain/wiki/pages/feedback_gate_verdict_over_exit_code.md`
for the 2026-04-11 near-miss incident.

### 5.4 Run contract (run / run-all only)

After the freshness gate passes with `pending_count>0`:

1. Safety gate:
   `${CODEX_LAB_PATH:-~/codex-lab}/experiments/brain-handoff-safety-gate.sh`
   FAIL: stop.
2. Start marker:
   `~/bin/brain-handoff-run-start "<lane-ids>"`
3. Execute lanes.
4. Immediately call closeout:
   `/brain-handoff-complete --completed <lane-ids> --summary "<result> || <next preference>"`
   Skipping step 4 causes the next handoff invocation to fail until closeout
   is performed.
5. Close parity:
   `bash ~/bin/brain-handoff-parity-gate close`
   On failure, treat closeout as incomplete and report the failing check.

### 5.5 Seed contract (seed-tasks only)

After the freshness gate passes with `pending_count>0`:

1. `~/bin/brain-handoff-seed [--lane L] [--max N] [--priority Pn] [--max-priority Pn]`
2. **Do not** call `handoff_run_start.sh`. **Do not** call
   `/brain-handoff-complete`.
3. Seed closeout:
   `~/bin/brain-handoff-closeout --summary "<seed result> || <next preference>"`
4. If seed mode opens run state, treat as contract violation and stop.

### 5.6 run-all behavior details

- Lane target set: all lanes with `status=pending` in the fresh
  `handoff-lanes.json`, optionally filtered by `--agent`, `--priority`, or
  `--max-priority`.
- `--agent` default: exported `BRAIN_AGENT` when available.
- Execute in registry order. After each lane: closeout then regenerate
  artifacts.
- Empty lane target set: resume active task owned by current agent first
  (force-reclaim if stale), then delegate queue selection to
  `brain-next-task` (see §8.2).
- Queue tasks are still claimed one-at-a-time through the task lifecycle
  in §4.
- Fail-closed: stop batch on first lane failure, emit resumable pointer.
- Never touch `status=completed` lanes.

### 5.7 Lane priority (P0--P3) -- distinct from task priority

| Label | Meaning |
|---|---|
| `P0` | Correctness, safety, repo hygiene, truth integrity. |
| `P1` | Core capability and operating-model upgrades. |
| `P2` | Operational verification and confidence-building. |
| `P3` | Polish and convenience. |

Default execution order: `P0 -> P1 -> P2 -> P3`. **This is not the
task-queue priority scale.** See §8.5 for the two-system disambiguation.

### 5.8 Credit discipline

- Keep output compact in `show` mode.
- Do not rerun completed lanes without regression evidence.
- Use `seed-tasks` when credits are low and queueing is preferred over
  execution.

## 6. Seat Behavior (shared)

### 6.0 Agent-agnostic first (pre-edit gate)

The Brain is an agent-agnostic AI OS. Every seat (claude-code, codex,
gemini, future seats) is a peer CLI into the same Brain. Content lives
in agnostic locations; seat-local files are thin wrappers.

**Pre-Edit Protocol — MANDATORY before any Edit/Write on seat-local paths
(`.claude/`, `.codex/`, `.gemini/`, `CLAUDE.md`, `AGENTS.md`, `GEMINI.md`,
per-seat command directories):**

1. **Does canonical content exist in an agnostic location?** Check
   `brain/wiki/` / `brain/governance/` / `brain/MANUAL.md` /
   `brain/core-knowledge/`. If NO → stop.
2. **Write canonical first.** Rule → `brain-wiki create feedback_<name>`.
   Tool → `brain/wiki/tools/<tool>.md`. Protocol → `brain/wiki/pages/` or
   `brain/governance/`. Runtime contract → `brain/MANUAL.md`.
3. **Update ALL seat surfaces together** — `MANUAL.md`, `CLAUDE.md`,
   `AGENTS.md`, `GEMINI.md`, per-seat command wrappers, digest entry.
4. **Self-check:** "If the other seats run this tomorrow, do they see
   the change?" If only the current seat sees it → not finished.

Enforced by `brain-closeout-gate` AGNOSTIC_DRIFT advisory (flags sessions
that edit seat-local paths without agnostic counterparts). Full rule at
[[feedback_agent_agnostic_first]]. See also
[[feedback_agnostic_default_not_claude_default]] (burden of proof is on
the lock-in claim) and [[feedback_seats_are_cli_not_api]].

Applies to: commands, protocols, checklists, runbooks, SOPs, rules,
procedural content. Does NOT apply to: SessionStart hook wiring, CLI
auth config, UI-only settings — but even then the wiki page describing
the mechanic lives in `brain/wiki/`.

### 6.1 Seat claim and arbitration

Only one agent owns the laptop seat at a time. Seat state lives at
`brain/ops/.brain-active-seat.json`. Fields: `seat`, `assistant`, `mode`,
`ts`, `source`. The bootstrap for your seat claims the file; the operator
(or the next seat's bootstrap) releases it.

`BRAIN_AGENT` environment variable is the current seat's canonical
identifier when exported. Handoff tools default `--agent` to `$BRAIN_AGENT`
when the flag is omitted.

### 6.2 Session-start gate (all seats)

Every seat, at session start, in this order:

1. **Get the operational brief.** Source differs by seat:
   - **Claude Code:** Read `brain/index/session-brief.md` — written fresh by
     bootstrap, ~375 tokens, contains verdict, gates, health, task counts,

### 6.3 Session-close pipeline

**Current canonical close paths (implementation truth):**

| Seat | Close command |
|------|--------------|
| Claude Code | `brain-meditate` |
| Codex | `brain-meditate` |
| Gemini | `brain-meditate` |

`brain-meditate` is the universal session-close orchestrator for every seat
(claude-code, codex, gemini). It runs the close pipeline in order with
fail-closed behavior on required stages. `brain-wrap-up` is a deprecation
stub that forwards to `brain-meditate` for backward compatibility.

**Expected post-close neutral state (all seats):**
- `brain/ops/.brain-active-seat.json`: `assistant=none`, `mode=standby`
- `brain/ops/.laptop-heartbeat.json`: `assistant=none`, `status=idle`, `remote_dm_lane=disabled`

| # | Stage | Behavior |
|---|---|---|
| 1 | `brain-state-check` | Read-only sanity. Aborts with exit 3 on DRIFT/CONFLICT. STALE continues. |
| 1b | `core-memory` | **HARD GATE.** Aborts if unresolved core-memory items exist. |
| 2 | `brain-memory-flush` | Extracts daily decisions/lessons from session digests. Soft-fail WARN. Skippable with `--skip-memory`. |
| 3 | `brain-memory-compile` | Compiles daily files into `compiled-insights.md`. Soft-fail WARN. Skippable with `--skip-memory`. |
| 4 | `brain-seed-audit` | Content/product seed check. Never blocks. WARN on gap. |
| 4.5 | `devlog-ensure` | Creates minimal devlog stub if none exists for today (local date). Populates from `--summary` + today's TASK_COMPLETE events. Soft-fail WARN — does not block; `closeout-gate` is the hard enforcement. |
| 5 | `brain-closeout-gate` | **HARD GATE.** See §6.4. |
| 6 | `brain-content-brief` | Opt-in only via `--with-content-brief`. Soft-fail WARN. |
| 6b | `session-digest` | Captures current seat's session conversations. Soft-fail WARN. |
| 6c | `seal-state` | **DURABILITY.** Non-interactive `git commit` and `git push`. |
| 6d | `checkpoint` | Creates a recovery checkpoint for the session. |
| 7 | `brain-session-end` | **Last.** Emits `SESSION_END` boundary marker and the wiki session entry. |

Exit codes:

| Code | Meaning |
|---|---|
| 0 | All required stages passed. `SESSION_END` logged. |
| 1 | Bad usage (unknown flag). |
| 2 | `closeout-gate` FAIL. Session-end NOT emitted. Fix issues and re-run. |
| 3 | `state-check` DRIFT/CONFLICT. Pipeline aborted before any stage. |
| 4 | A required stage broke mid-pipeline. Session-end NOT emitted. |

Flags (all seats):

| Flag | Default | Purpose |
|---|---|---|
| `--agent <name>` | optional | Seat identifier (`gemini`, `codex`, `claude-code`, ...). Auto-detected from `BRAIN_AGENT` when omitted. |
| `--summary "text"` | empty | Forwarded to `brain-session-end` and written to the wiki session entry. |

| `--skip-memory` | false | Skip `brain-memory-flush` and `brain-memory-compile`. |
| `--dry-run` | false | Read-only dry-run. See "No mutation in --dry-run" below. |
| `--json` | false | Emit a single JSON object with per-stage status instead of human-readable lines. |

runs exactly once per session and removes `.active-session.json`.

**No mutation in `--dry-run`.** Every stage runs in dry-run mode where
supported; `session-end` is skipped entirely. Useful for CI smoke tests
and for verifying the close pipeline without sealing the session.

Rollback procedures:

- Memory stages: delete latest date entries in
  `brain/ops/daily/YYYY-MM-DD.md` and `brain/ops/compiled-insights.md`, or

- `session-end`: delete the wiki session entry at
  `brain/wiki/sessions/YYYY-MM-DD-HHMM-<agent>-session.md` and remove the

  `.active-session.json` marker can be restored from the session-start
  marker if retained.

### 6.4 brain-closeout-gate (shared hard gate)

`brain-closeout-gate` is called by every close path. Fail-closed on:

1. Missing same-day devlog.
2. Stale or missing usage telemetry.
3. Changed `tools/bin/brain-*` scripts without a matching
   `brain/wiki/tools/<tool>.md` page and an entry in
   `brain/wiki/tools/MAP.md`.
4. **Unresolved `CORE_MEMORY:` queue items**
5. **Uncommitted changes at seal-state** — `brain-meditate` stages and commits all dirty files before finishing. A meditation that exits without a commit when changes exist is a failure, not a skip.

**Truncation protection (pre-commit hook).** The git pre-commit hook blocks any commit that reduces line count on append-only surfaces: `brain/ops/events.*`, `work/logs/*.jsonl`, `work/logs/*.log`, `work/logs/devlog/*.md`, `work/logs/wins-log.md`, wiki digest pages. No seat can accidentally wipe these surfaces — the hook catches it at commit time regardless of which agent made the change. (gate section 4b). Previously
   enforced only by `brain-meditate`; as of the GAP-006 fix, enforced by
   every close path via `brain-meditate` inheritance. If
   `brain-core-memory-status` is missing from `PATH`, the gate emits a
   loud WARN (`CORE_MEMORY_TOOL_MISSING`) rather than silently skipping.

The MEMORY_UNMAPPED WARN (memories without matching wiki pages) also
surfaces here, as a warning -- not a hard block.

- `brain-session-end` -- invoked as the final stage of `brain-meditate`.
  Calling it directly skips memory flush/compile and the closeout gate,
  leaving session close incomplete.

## 8. Execution Flow and Advisory Tools

### 8.1 Canonical sequence

```
resume --> state check --> advisory --> human auth --> execute --> verify --> close
```

Every seat runs this sequence. Only the invocation at `resume` and `close`
differs (§7).

### 8.2 brain-next-focus vs brain-next-task

These are **two distinct tools**. Keep them straight.

| Tool | Role | When to use |
|---|---|---|
| `brain-next-focus` | **Advisory.** Recommends a focus for the next working block based on priority queue, freshness, and deferred work. Read-only. | Planning step between "resume" and "human auth". Do not auto-run without an operator prompt. |
| `brain-next-task` | **Executor selector.** Picks the next executable task from the queue. Handles agent routing, deferred-lane filtering, priority selection. Called by handoff `run-all` empty-target fallback. | Right before `brain-task-claim` / `brain-task-execute`. |

The "Prepared Focus" line in `session-resume.md` is output from
`brain-next-focus`. Authorizing a prepared focus is an operator act; the
advisory never auto-executes.

### 8.3 Continuation signals

After `brain-task-complete`, the tool emits a continuation signal. How to
act on it depends on the **execution mode**.

**Execution modes:**

| Mode | How it starts | CONTINUE behavior |
|---|---|---|
| **Single-task** | Operator assigns one task ("do this task", "run TASK-XXX") | Report completion, return control to operator. Do not auto-proceed. |
| **Batch** | Operator explicitly authorizes multiple tasks ("run the queue", "do these 5 tasks", "keep going", handoff run-all) | Auto-proceed: run `brain-next-task --dry-run`, state the next task, proceed. |

Batch authorization is **operator-initiated** -- never assumed from task
system signals alone. A CONTINUE signal does not grant batch mode; the
operator's words do.

**Signal table (applies in both modes unless noted):**

| Signal | Meaning | Action |
|---|---|---|
| `CONTINUE` | More work in queue. | **Batch mode:** auto-proceed to next task. **Single-task mode:** report completion and return control to operator. |
| `CODEX` | Hand off to another seat for independent verification. | Prepare a handoff packet (task state, open questions, work artifacts) and start the receiving seat. |
| `STOP` | Session should end. | State the reason and the recommended next action. Do not continue into another task. |

**Post-task confirmation rule (enforced):**
- `brain-task-confirm` now requires verification evidence before clearing
  `.pending-audit` for P1/P2 tasks or governance-touching tasks.
- Accepted evidence: a verification artifact referencing the task
  (canonical: `brain-review-dispatch <TASK-ID> --type verification`;
  compatibility wrapper: `brain-task-handoff <TASK-ID> --type verification`).
- Emergency/manual bypass requires explicit operator intent:
  `brain-task-confirm <TASK-ID> --allow-unverified --reason "<why>"`.

**Chain depth >= 4 triggers a mandatory operator check-in** regardless of
signal or mode.

### 8.4 Autonomous execution rules

Autonomous execution requires explicit operator instruction and is capped:

- **P3 priority only.**
- **Maximum 4 tasks per batch.**
- **Stop** on first of: `STOP`, `DRIFT`, `CONFLICT`, any task failure.

**Control-First (ADR-0003).** Autonomy is granted, not assumed.

### 8.5 Priority systems -- do not conflate

The Brain runs two priority scales. They look similar; they are not.

| Scale | Labels | Surface | Meaning |
|---|---|---|---|
| **Task priority** | P1, P2, P3, P4 | Task queue (`brain/ops/tasks/`) | P1 = blocking revenue/production. P2 = ship today. P3 = scheduled. P4 = nice-to-have. |
| **Handoff lane priority** | P0, P1, P2, P3 | Handoff system (`codex-lab/handoffs/`) | P0 = correctness/safety. P1 = core capability. P2 = verification. P3 = polish. |

A task labelled P1 and a lane labelled P1 do not mean the same thing.
Never promote or demote items between scales without rewriting the label.
State the priority scale explicitly when referencing a level in operator
communication ("P1 task" vs. "P1 lane").

### 9.9 Operational discipline (shared)

- **Ask before building.** When told to "look for better solutions" or
  similar open-ended exploration, **stop and ask** before building. Do not
  interpret exploration prompts as build prompts.
- **Use `brain-*` wrappers, not ad-hoc shell chains.** Operations must be
  deterministic and reviewable. If a brain-* tool does not exist for a
  task, propose one; do not assemble a pipe fest of `sed`/`awk`/`jq` in
  its place.
- **Checkpoint before risky ops.** Run `brain-checkpoint "reason"` before
  operations that could be hard to reverse (schema changes, large refactors,
  destructive file moves). Claude Code auto-checkpoints ~every 30 min via
  the PreCompact hook; other seats must run it manually.
- **Never run `brain-task-execute` in background.** It produces execution
  context output the agent must read immediately to know what to do.
  Running it with `run_in_background: true` causes stale BackgroundTaskComplete
  notifications that arrive after the agent has already moved on. Always
  call brain-task-execute synchronously (foreground). This applies to all
  seats. Claude Code has no `BackgroundTaskComplete` hook type, so there
  is no automatic remedy — prevention is the only fix.
- **Verify fixes against synthetic surfaces first.** Before running a fix
  against any production surface, build a standalone regression test first
  (`mktemp -d` scratch dir, synthetic JSON inputs, explicit `--dry-run`
  modes). Production-surface verification is a fallback only — "reversible"
  does not authorize the initial touch if a scope directive restricts it.
  If live verification is truly needed, state the target, expected side
  effect, and rollback plan, then wait for explicit approval. See
  `brain/wiki/pages/feedback_verify_fixes_against_synthetic_surfaces.md`
  for the incident that generated this rule.

### 9.10 Execution routing: deterministic first (applies_to: all)

Route work through deterministic code before AI inference. If a task
can be solved by rule-based code (scripts, hooks, gates, validators),
that code runs first. AI inference handles only what code cannot:
ambiguity, judgment, creativity, context-dependent decisions.

**Layered model:** Layer 1 = deterministic code (100% reliable, zero
tokens). Layer 2 = AI inference (probabilistic, token-expensive).
Layer 1 runs first; Layer 2 handles the remainder.

**Implementation:** New tools must extract deterministic logic into
code. Behavioral rules that describe automatable checks should become
scripts. Gates check and pass/fail -- they never reason. Cross-agent
dispatch uses deterministic routing; query content is AI.

Full policy: `brain/governance/execution-routing-policy.md`.

### 9.11 Agent lifecycle: bootstrap, session, exit (applies_to: all)

Agent startup is two phases: **onboarding** (first seat, registers agent
and capabilities) and **session** (every session, enforces governance and
memory). Sub-agent selection across agents is deterministic based on task
type and capability profiles, not arbitrary.

Session exit has two modes: **full shutdown** (seal and stop), **agent
swap** (seal, spawn next agent, verify handoff, terminate). Agent swap lets
the operator or system
pick the next agent (Claude Code, Codex, Gemini CLI) based on task
requirements and rate limit headroom.

Full policy: `brain/governance/agent-lifecycle-policy.md`.

### 9.12 Brain infrastructure: machine-first design (applies_to: all)

For internal brain infrastructure and tooling, **prefer machine-usable
structure over agent-process work** when code can solve the problem cleanly.

Rules:

1. **Machine-readable outputs.** Indexes, state files, and classification
   structures must be machine-consumable (JSON/YAML), not prose-only.
   Human-readable MAP files and prose docs are allowed alongside them but
   not as a substitute.

2. **Explicit metadata over inference.** If a property of a page or artifact
   can be stated explicitly (page_type, topic_bucket, is_hand_authored),
   state it in a generated index. Do not rely on agents to infer it from
   context at query time.

3. **Generated indexes over manual tracking.** When a set of files needs to
   be navigable, generate an index from source of truth (file names,
   frontmatter, MAP.md sections) rather than maintaining it by hand or
   relying on fuzzy search alone.

4. **Code over agent-process where code is the better resolution.** If a
   classification, check, or structural invariant can be enforced by a
   script, enforce it there. Reserve agent judgment for decisions that
   genuinely require judgment.

5. **Working mode: reason like a debugger/compiler/orchestrator.** When
   doing brain infrastructure work, break problems into explicit states,
   invariants, inputs, outputs, and transitions. Prefer executable checks,
   deterministic transforms, and generated artifacts. Minimize reliance on
   agent memory or process when code can carry the logic cleanly.

6. **Reporting discipline: result-only output to operator.** Do not narrate
   internal steps for human readability during infrastructure work. Report
   only: what changed, what state is now true, what blocked, what the next
   decision or action is. Prose narration is for customer-facing and
   human-directed deliverables — not for internal brain ops.

**Scope:** this applies to brain infrastructure: task system, wiki indexes,
state files, tool docs, governance structure. It does not apply to
customer-facing content, product copy, or anything explicitly human-facing
where prose and judgment are the right tool.

{sources: 2026-04-13-operator-directive, TASK-2026-04-13-014, 2026-04-13-operator-directive-working-mode}

### 9.13 Permission modes and authority stops (applies_to: all)

`bypassPermissions` / yolo mode removes **Claude Code tool-call prompts** —
the per-call "do you want me to run this?" confirmations. It does **not**
remove or weaken any of the following:

- Brain governance requirements (§9.4 write surfaces, §9.2 destructive ops)
- Operator-decision gates (publishing §9.3, destructive ops §9.2)
- Hard workflow stops (`brain-closeout-gate`, `brain-task-execute` audit
  marker, `VERDICT_GATE`, `CORE_MEMORY` queue, chain-depth check-in)
- Mandatory operator approvals listed anywhere in this MANUAL

If a rule says "requires operator interaction," "explicit approval," or
"mandatory stop," that requirement is **not satisfied by running in bypass
mode**. Bypass is a tool-friction reducer, not an authority override.

This distinction applies to every seat. Seats that run in bypassPermissions
by default (e.g., Claude Code via settings.json) are held to the same
governance requirements as interactive sessions.

### 9.16 Sub-agent dispatch — custom system over native (applies_to: all)

**Always use `brain-subagent-dispatch` (or `brain-subagent-dispatch --profile <id>`) instead of the native agent-spawning mechanism (Claude Code `Agent` tool, direct CLI calls, etc.) when delegating work to another agent.**

The custom dispatch system provides:
- Execution profile routing (correct model per task type)
- Capability health checks before dispatch
- Model registry enforcement (`auto_dispatch_forbidden` guard, 1M protection)
- Event logging (`SUBAGENT_DISPATCH` in `events.log` + `events.jsonl`)
- Deterministic fallback order across providers

**Native `Agent` tool is allowed only when:**
1. The task is a one-off exploratory query with no Brain state implications, OR
2. `brain-subagent-dispatch` is unavailable or broken (document why in devlog)

When in doubt: `brain-subagent-dispatch --profile <closest-match> --question "..."`.

**No fitting profile? Make one first.** The dispatch center is an expanding dressing room — profiles are outfits. When no profile captures the reasoning stance needed, create one before dispatching: write the preamble (`ai/prompts/core/<name>-system-preamble.md`), add to `execution-profiles.json`, verify with `--dry-run`. See `brain/wiki/tools/execution-profile-creation.md` for the full SOP.

**Profile trigger rules — use these reflexively, not as special actions:**

| Trigger | Profile | Command pattern |
|---|---|---|
| About to ship P0/P1 or irreversible change | `adversarial-critic` | `brain-subagent-dispatch --profile adversarial-critic --question "Red-team this plan: ..." --files <spec>` |
| Completing a task that touches governance, architecture, or multi-system wiring | `auditor-skeptic` | `brain-subagent-dispatch --profile auditor-skeptic --question "Verify: ..." --files <changed-files>` |
| Starting a task with broad architectural impact | `systems-architect` | `brain-subagent-dispatch --profile systems-architect --question "Review design for invariant preservation: ..."` |
| Need to understand unknown territory before executing | `research-mapper` | `brain-subagent-dispatch --profile research-mapper --question "Map: ..." --files <relevant-areas>` |
| Any audit, gap-analysis, or verification pass | `auditor-skeptic` or `repo-auditor` | default to auditor-skeptic for opinion work; repo-auditor for read-only scan |
| Large codebase sweep or session log analysis | `long-context-analysis` | routes to Gemini automatically |
| P3/P4 implementation task, no critical path | `rapid-implementer` | omit --profile; auto-routes to cheapest healthy agent |

Stance profiles (`auditor-skeptic`, `adversarial-critic`, `systems-architect`, `research-mapper`) inject a system-level reasoning preamble — the target agent operates in that reasoning mode for the duration of the call. These are multi-provider by default (Gemini/Codex for auditing stances, Claude Opus for architecture).

**Skill drawer** — layer add-ons on top of any profile at dispatch time:
- `--skills cite-sources,verbose` — behavior flags (cited claims, full reasoning chain)
- `--skills opus` — model override (upgrade to Opus for this call only)
- `--bundle brain-critic` — named bundle (read-only + cite-sources + verbose)
- `--chain p1,p2,p3` — sequential pipeline; each step's output feeds the next
- `--auto-profile --task TASK-ID` — auto-select profile from task's `task_type` field
- `--browse` — display all profiles + skills without dispatching

See `brain/wiki/tools/skill-registry.md` for full skill/bundle list and chaining SOP.
See `brain/wiki/tools/brain-subagent-dispatch.md` for full dispatch usage.

**Sub-agents are stateless executors.** A dispatched sub-agent has no
session ownership and no lifecycle authority. Lifecycle commands —
`brain-meditate`, `brain-checkpoint`, `brain-handoff`, and the
`brain-session-end-commit` Stop hook — refuse to run when
`BRAIN_DISPATCH_ACTIVE=1` is set in the environment, which dispatch
wrappers export. The operator's session owns lifecycle; sub-agents
return a result and stop.

**Profile-scoped tool/context isolation (framework concept).** Execution
profiles can declare a minimal tool surface for the sub-agent — which
MCP servers are reachable, which CLI extensions to load, which working
directory to neutralize project-scoped hooks, and a runtime policy file
that excludes denied tools from the agent's tool registry entirely (both
a security and a context-cost win). The mechanism is generic; specific
profile contents are operator-defined and live outside the public
framework.

### 9.14 1M context model authorization (applies_to: all)

Models with `"context_window": 1000000` (1M variants, e.g. `claude-opus-4-6-1m`,
any future `claude-sonnet-*-1m`) are billed as **extra usage** and must never
be auto-selected, auto-dispatched, or implicitly invoked.

**Rules:**

- Brain systems must not dispatch to a 1M model without explicit operator instruction
  in the current session.
- Execution profiles must not list a 1M model as a fallback for claude-code.
  Long-context work defaults to Gemini. If Gemini is unavailable, operator
  selects the model manually.
- `auto_dispatch_forbidden: true` in model-registry.json is the machine-readable
  marker. Any dispatch path that reads model-registry must respect this field.
- The brain **may** suggest a 1M model as an option when a task clearly requires
  it, but must label it: _"Requires explicit operator authorization — extra usage."_
- Manual `/model` selection by the operator is allowed at any time. That is
  the intended path for intentional 1M use.
- `bypassPermissions` / yolo mode does NOT override this rule (§9.13).

### 9.15 Model-role policy (applies_to: all)

Model choice is a cost/capability tradeoff, not a habit. Use the weakest
model that is competent for the work class. The routing infrastructure
(execution-profiles.json, model-registry.json) is the machine-readable form
of this policy.

| Model | Role | Brain use |
|---|---|---|
| `claude-haiku-4-5` | Cheap/fast, low judgment | Well-scoped mechanical tasks: manifest updates, index rebuilds, frontmatter edits, classification. `light-automation` profile. NOT for governance, architecture, or operator-judgment decisions. |
| `claude-sonnet-4-6` | Default seated agent | Most queue tasks, interactive sessions, operator conversation, implementation work, self-assessment. `rapid-implementer`, `frontend-designer`, `discord-systems` profiles. |
| `claude-opus-4-7` | Careful/critical path | Multi-system refactors, governance changes, critical path edits. `careful-refactorer` profile. Operator must authorize each task. |
| `claude-opus-4-7-1m` | 1M context, extra billing | Explicit operator authorization required. Never auto-dispatched. See §9.14. |
| `gpt-4.1` (Codex) | Sandboxed audit/analysis | Read-only investigation, structured verification. `repo-auditor` profile. |
| `gemini-2.5-pro` | Long-context, multimodal | Full-repo sweeps, large document analysis. `long-context-analysis` profile. |

**Haiku as seated agent:** Viable for sessions where all tasks are fully
pre-specified P4 work with no governance surface. Requires model escalation
to be wired (TASK-2026-04-13-007) so Sonnet/Opus can be summoned for
harder steps. Not viable as the interactive seat for architecture, governance,
or open-ended operator sessions.

**Bias to avoid:** Do not use Opus for tasks that Sonnet handles well.
Do not use Sonnet for tasks that Haiku handles well. Route on evidence.

**Resource-pressure thresholds and per-model-class guidance** are configurable at:
`brain/ops/resource-pressure-policy.json` — operator-adjustable without code edits.

The policy file contains:
- `thresholds.warn_pct` / `thresholds.high_pct` (defaults: 70 / 85)
- Per-model-class guidance at each level: `haiku`, `sonnet`, `opus`, `1m`
- Actions: `proceed` (no advisory) / `advisory` (surface note, operator decides) / `defer` (strong advisory) / `hard_boundary` (non-negotiable)
- `hard_boundaries` section documents what is NOT adjustable and why

| Pressure | Haiku | Sonnet | Opus | 1M |
|---|---|---|---|---|
| ok | proceed | proceed | proceed | hard boundary |
| warn | proceed | proceed | advisory | hard boundary |
| high | proceed | advisory | defer | hard boundary |

**Key distinction — advisory vs hard boundary:**
- Advisory means: signal is surfaced, operator intent overrides. Explicit operator authorization satisfies any advisory at any level.
- Hard boundary means: cannot be satisfied by pressure policy alone. Requires the specific authorization defined in §9.13 / §9.14.

`brain-next-focus` surfaces the current pressure level and per-task profile in
its output. The signal comes from the authoritative statusline snapshot (5h and
7d rate-limit windows) and is **claude-code-only** — it does not represent
Codex or Gemini headroom.

`brain-resume` also surfaces claude-code 5h/7d by reading `usage-snapshot.json`
as a fallback when the JSONL adapter returns null rate limits.

**Provider signal availability (summary):**
- Claude Code: 5h/7d authoritative via statusline → `usage-snapshot.json`
- Codex: 5h/7d authoritative via session JSONL token\_count events, but only when snapshot is explicitly refreshed (`brain-seat-usage-snapshot --agent codex`)
- Gemini: no rate-limit signal available at all

See `brain/ops/usage-schema.json` for the full per-provider signal inventory.

## 10. What Not to Edit

Protected surfaces are listed in §9.4. This section adds three rules that
have bitten seats in the past.

### 10.1 Bin tools and the wiki-mapping requirement

Any edit to `~/bin/brain-*` requires, in the **same
session**, a matching wiki tool page at `brain/wiki/tools/<tool>.md` **and**
an entry in `brain/wiki/tools/MAP.md`. Enforced at close time by
`brain-closeout-gate`. Editing a bin tool without updating the wiki will
block session close with `BIN_TOOL_WIKI_MISSING`.

**Full creation canon:** When building any new tool — any domain — follow `brain/wiki/tools/tool-creation-canon.md`. Required: script in `scripts/bin/`, `chmod +x`, `~/bin/` symlink, wiki page with all six required sections, registry regeneration (`brain-agent-registry-generate`), wiki index rebuild (`brain-wiki rebuild-index`), and cross-links on related tool pages. The wiki page must have `## SOP`, `## Usage`, `## What This Fixes`, `## Combines With`, and `## Wired Into` — not just SOP. See Layer 5 of post-task assimilation audit for the gate check.

### 10.2 Crontab

Never pipe crontab through `sed` or `awk`. Always use `brain-cron-update`.
See §9.5.

### 10.3 Auto-generated truth files

Do not hand-edit. The main offenders:

| File | Generator |
|---|---|
| `brain/index/session-resume.md` | `brain-resume` |
| `brain/index/session-start-contract.compact.md` | `brain-resume` |
| `brain/ops/derived/system-state.md` | `brain-resume` |
| `brain/ops/derived/state-verdict.json` | `brain-state-check` |
| `brain/ops/derived/freshness.json` | freshness tools |
| `brain/ops/compiled-insights.md` | `brain-memory-compile` |

Regenerate via the correct tool. Manual edits will be overwritten.

### 10.4 Non-bin capability registration

Bin tool changes are gated (§10.1). Non-bin capabilities are not mechanically
gated, but carry the same registration obligation. A capability is **not
assimilated** until it is documented, registered, and discoverable.

| Capability type | Required on creation |
|---|---|
| `.claude/commands/*.md` (skills) | Entry in `brain/wiki/tools/MAP.md` + wiki tool page at `brain/wiki/tools/<name>.md` **with SOP section**. Reference in CLAUDE.md or MANUAL if it changes session behavior. |
| `scripts/bin/brain-*` (bin tools) | Wiki tool page at `brain/wiki/tools/<name>.md` **with SOP section** + MAP.md entry (gated by closeout). |
| `settings.json` hooks or model changes | Entry in MANUAL §7.1 (Claude seat deltas) and/or the relevant close path doc if it affects session lifecycle. |
| New MANUAL sections or CLAUDE.md rules | Mention in the session devlog. No separate wiki page required unless the rule is complex enough to warrant one. |
| New cron entries | Covered by `brain-cron-update` (§10.2). |

**SOP requirement (HARD).** Every `brain/wiki/tools/<name>.md` page MUST contain a `## SOP` section. No SOP = not assimilated. Format:
```
## SOP
**Prerequisites:** ...
1. Step
2. Step
**Expected output:** ...
**If it fails:** ...
```

**"Built" does not mean "adopted."** After any session that adds a non-bin
capability, the post-task self-assessment and meditate review must confirm
the capability is registered and a fresh agent could discover and use it.
If it is not registered, it is invisible infrastructure — effectively lost.

## 12. Promotion Path: Memory, Wiki, Core Memory

### 12.0 Active learning (HARD)

**Write memory / wiki / seed NOW on error, discovery, or decision, before
the next tool call.** Do not batch lessons for end-of-session meditation.
Do not rely on "I'll remember this and write it up later". The brain's
value comes from persisting lessons at the moment they are still concrete.

| Discovery | Destination |
|---|---|
| Behavioral correction from operator | Memory + wiki page (§12.2) |
| Content or product idea | `work/drafts/queue/seeds.md` (inline append) |
| Lesson or pattern worth preserving | Wiki page in `brain/wiki/pages/` |
| Project state change | Wiki system page or `active-context.md` |
| Unclear category | Inbox stub in `work/inbox/` |

Memory is **not complete** until a matching wiki page or inbox stub
exists. See §12.2 for the full completion rule. Closeout-gate will WARN on
any memory that has no matching wiki page.

**Agent-agnostic memory (HARD).** All memory content belongs in the Brain wiki — not in any agent's local memory store. Local memory files (Claude, Gemini, Codex) must be **pointer-only**: frontmatter with a `wiki:` field, no body. A rule written only in Claude's memory is invisible to Gemini and every future agent. The wiki is the single durable surface all agents share.

### 12.0.1 Issue and error documentation (HARD)

**Any time work hits an error, unexpected behavior, or non-obvious workaround — document it in the relevant wiki page before continuing.** Do not defer to end-of-session. Do not only add it to the devlog.

**Which wiki page:**

| Issue type | Destination |
|---|---|
| Tool, script, or CLI misbehavior | `brain/wiki/tools/<tool>.md` — add a "Gotchas" or "Known issues" section |
| Infrastructure / OS / environment quirk (WSL2, inotify, process signals) | `brain/wiki/pages/<topic>.md` — create if it doesn't exist |
| System or service failure mode | `brain/wiki/systems/<system>.md` or the relevant system page |
| Pattern that will recur across tools | `brain/wiki/pages/` as a standalone reference page |

**Format:** One heading per issue. State the symptom, the root cause, and the fix. Include the exact command or code that works. See `brain/wiki/pages/inotify-wsl2.md` as the canonical example.

**When a fix changes a script:** Update the wiki page AND the script in the same commit. The wiki page is the explanation; the script is the enforcement. Both must stay in sync.

### 12.1 Layered persistence model

| Layer | Target | Purpose |
|---|---|---|
| identity | `brain/core-knowledge/identity.md` | Only if identity-relevant (requires `--identity`). |
| core-signal | `brain/core-knowledge/core-signal.md` | Non-negotiable canon. |
| memory | Seat memory store (§7) | Session-persistent behavioral rule. |
| wiki | `brain/wiki/pages/*.md` or `brain/wiki/systems/*.md` | Canonical knowledge surface. |
| journal | `brain/ops/events.log` | Timestamped promotion event. |

### 12.2 Memory-completion rule

"Memory saved" is **not** complete until:

1. Memory file exists in the seat's memory store.
2. Matching wiki page exists in the correct folder:

   | Prefix | Wiki folder | MAP file |
   |---|---|---|
   | `feedback_*` | `brain/wiki/pages/` | `RULES.md` |
   | `project_*` | `brain/wiki/systems/` | `SYSTEMS.md` |
   | `reference_*` | `brain/wiki/systems/` | `SYSTEMS.md` |
   | `user_*` | `brain/wiki/people/` | `people/MAP.md` |

3. MAP file references the new wiki page.
4. Both paths printed in output: `memory: <path>` and `wiki: <path>` (or
   `inbox: <path>` if wiki write is blocked).

If wiki write is blocked because the category is unclear or requires
approval, create an inbox stub at `work/inbox/` and note the stub path
instead of the wiki path.

`brain-resume` runs a memory-wiki consistency check on session start and
reports `Memory: N mapped, M unmapped`. Unmapped memories are flagged in
resume output. Closeout-gate emits `MEMORY_UNMAPPED` WARN for any unmapped
memory at session close.

### 12.3 Core memory promotion

Operator messages containing `CORE_MEMORY:` are mandatory promotion
candidates. They cannot be skipped by model judgment.

**Surfaces:**

- Queue: `brain/ops/core-memory/queue.jsonl`
- Resolved: `brain/ops/core-memory/resolved.jsonl`
- Each item: `id`, `ts`, `source`, `statement`, `tags`, `identity_relevant`,
  `status`.

**Tools:**

| Tool | Purpose |
|---|---|
| `brain-core-memory-capture` | Queue a statement for mandatory promotion. |
| `brain-memory-promote-core` | Perform the 5-layer promotion with PoE evidence. |
| `brain-core-memory-resolve` | Mark item resolved after PoE-verified promotion. |
| `brain-core-memory-status` | Show unresolved queue items. |

**Inline marker rule.** When an operator message contains
`CORE_MEMORY: <statement>`, the current agent must immediately call
`brain-core-memory-capture` with the statement. This ensures the queue is
populated before any model-judgment skip can occur.

**Gate.** Closeout-gate section 4b hard-blocks session close on unresolved
core-memory queue items. This gate is shared by every close path via
`brain-meditate` (§6.3 and §6.4). `brain-resume` shows the unresolved count
and top items in the Gate Summary.

### 12.4 Resolution flow

```bash
# 1. Capture
brain-core-memory-capture --statement "..." --tags topic
# (tool returns capture_id)

# 2. Promote (all 5 layers + PoE)
brain-memory-promote-core "statement" --tag topic [--identity] [--memory-name name]
# (tool returns run_id with PoE verdict)

# 3. Resolve
brain-core-memory-resolve --id <capture_id> --run-id <run_id>
```

An item can only be resolved after `brain-memory-promote-core` returns
PoE verdict `PASS`.

### 12.5 PoE evidence for promotions

Every multi-layer promotion emits a proof-of-execution run:

- `brain-proof-start` at promotion start
- `brain-proof-log` for each layer write
- `brain-proof-verify` at end with verdict

### 12.6 What not to save

- Code patterns, architecture, file paths already derivable from the repo.
- Git history, who-changed-what (use `git log` / `git blame`).
- Debugging solutions already captured in commit messages.
- Anything already documented in this manual or seat governance files.
- Ephemeral task details (use the task system, not memory).

## 13. Common Workflows

Concrete paths for the four workflows that matter most.

### 13.1 Start session

1. Seat-specific bootstrap (§7). Claude auto-runs via hook; Codex runs
   canonical launcher command: `brain-claude`, `brain-codex`, or
   `brain-gemini`.
2. Claude only: read `brain/index/session-entry-contract.md`
   (hard-enforced by brain-sessionstart-handoff-gate).
3. Brief from the bootstrap/brain-resume stdout tail — do NOT Read
   `brain/index/session-resume.md` by default. See §6.2 and
   [[feedback_resume_brief_from_tail]].
4. Read `brain/index/session-start-contract.compact.md` (Tier-0 startup
   contract). Full MANUAL/digest reads are on-demand only.
5. Skip `brain/ops/derived/system-state.md` unless the tail shows an
   anomaly needing the full service/agent table.
6. Check Prepared Focus line (present in the tail). If fresh and Ready,
   present for operator authorization.
7. Present briefing to operator: verdict, health, queue, P1/P2 task items,
   freshness, prepared focus, incidents, suggested action.
8. Check `brain/index/active-context.md` freshness (>30 min: offer
   refresh).
9. Run `brain-ingest-discord --since 24h` if Discord surface is relevant.
10. Close with a concrete recommended next action, not an open question.

### 13.2 Pick up and execute a task

1. Operator authorizes a specific task or prepared focus.
2. `brain-task-claim <TASK-ID> <agent> [estimate_minutes]`
3. Read the task file in full before starting.
4. State priority before executing. P1 task must be logged immediately.
5. For AI-owned contracted tasks: start a proof run (§4.6).
6. Execute. Admit follow-ups via `brain-task-quick` as they emerge.
7. Update `system-state.md` if the task caused significant state change.
8. Session details -> devlog. Durable changes -> `active-context.md`.
9. `brain-task-complete <TASK-ID> --by <agent> --output "summary" --notes "..."`
10. Act on the continuation signal per §8.3 execution mode rules (single-task: return control; batch: auto-proceed).

### 13.3 Handle deferred or blocked work

1. If verdict is `DRIFT` or `CONFLICT`: **do not** mutate state.
   Investigate the drift source first (§14.1).
2. If a task becomes blocked mid-execution: set `status=blocked` via the
   tool path, not manually. Admit a follow-up task for the unblock work.
3. If a lane is deferred during a `run-all`: record the deferral reason in
   the lane `notes` field, continue to the next lane, emit a resumable
   pointer at batch end.

### 13.4 Close session

Close when **any** of: queue empty, chain depth >= 4, tasks need human
approval, operator signals end.

| Seat | Command |
|---|---|
| Claude Code | `brain-meditate` |
| Codex | `brain-meditate` |
| Gemini | `brain-meditate` |

All three paths call `brain-closeout-gate` as the HARD GATE. If closeout
fails with exit 2, fix the flagged issue and re-run the close command. The
session stays logically open until `brain-session-end` emits the boundary
marker.

**Do not call `brain-session-end` directly.** It is the last stage of the
close pipeline.

At session close, the seat output should include a "Memory Used" and
"Memory Added / Promoted" summary. Any new memories created this session
trigger the completion check described in §12.2 -- unmapped memories
surface as closeout-gate WARN.

## 14. Failure Modes and Recovery

Symptom -> diagnose -> fix. Operational, not prose.

### 14.1 State verdict is DRIFT or CONFLICT

**Symptom.** `brain-state-check` returns `DRIFT` or `CONFLICT`.
`brain-meditate` stage 1 aborts with exit 3. Handoff system refuses to run.

**Diagnose.**

```bash
brain-state-check
cat brain/ops/derived/state-verdict.md
```

### 14.3 brain-meditate exits 2 (closeout-gate FAIL)

**Symptom.** Close path reports `closeout-gate FAIL`. `SESSION_END` not

**Diagnose.**

```bash
brain-closeout-gate --json | jq .issues
```

**Fix by issue code.**

| Issue code | Fix |
|---|---|
| `MISSING_DEVLOG` | Write today's devlog at `work/logs/devlog/YYYY-MM-DD.md`. |
| `USAGE_TELEMETRY_STALE` | Refresh usage telemetry (see `brain/wiki/tools/brain-usage-*`). |
| `BIN_TOOL_WIKI_MISSING` | For each listed `brain-*` tool: create `brain/wiki/tools/<tool>.md` and add an entry in `brain/wiki/tools/MAP.md`. |
| `CORE_MEMORY_UNRESOLVED` | For each queue item: `brain-memory-promote-core "<statement>" --tag <topic>` then `brain-core-memory-resolve --id <capture_id> --run-id <proof_run_id>`. |

| `MEMORY_UNMAPPED` | For each listed memory: create the matching wiki page (§12.2) and update the MAP. Warn-level, not a hard block. |

### 14.9 Post-compaction / post-/clear restore

`applies_to: claude-code`

**Symptom.** Claude Code context was compacted mid-session, or the operator
ran `/clear` and asked to resume from a checkpoint. In-memory state is
partially or fully lost.

**Fix — exact restore sequence.**

1. Read the named checkpoint file. Resolve to `work/inbox/checkpoints/<filename>.md`.
   This is the **only** checkpoint location — do not probe `brain/ops/checkpoints/`
   or any other path. If the operator specifies just a filename (no path), prepend
   `work/inbox/checkpoints/`. If no filename is given, use the newest file under
   `work/inbox/checkpoints/` by filename sort.

2. Trust from the checkpoint without re-reading:
   - git state (branch, dirty files, ahead of remote)
   - reason / note
   - `active_lane`, `active_task`, `active_proof_run`
   - `pending_decisions`, `next_step`, `stop_conditions`

3. Re-read live — these are never trusted from the checkpoint:
   - `brain/index/session-resume.md` — generated state; checkpoint focus is frozen
   - `brain/ops/derived/state-verdict.json` — mutation gate must be current;
     read the `status` field (not `verdict`) for the gate value (OK/STALE/DRIFT/CONFLICT);
     if older than 5 minutes, emit WARN before proceeding
   - `brain/ops/tasks/active/` — verify the checkpoint's `active_task` is still
     claimed; task state can change between checkpoint and restore

4. **Do not** re-read `brain/index/session-entry-contract.md` on restore. That
   surface is gated to SessionStart only. Restore is a mid-session recovery,
   not a new session.

5. Present the restore briefing in this exact format:
   ```
   Restored from [filename] | Lane: [active_lane] | Task: [active_task or none] | Next: [next_step] | Pending: [pending_decisions] | Verdict: [OK/WARN/DRIFT]
   ```

6. Proceed with the stated `next_step` unless the operator corrects it.
   If any Restore Contract field reads `[unset — operator must confirm]`,
   ask the operator to confirm that field before proceeding.

**Note on SessionStart.** SessionStart runs `brain-agent-bootstrap`
automatically at true session open. `/clear` does not re-trigger
SessionStart. This restore sequence is the complete authority read for
post-`/clear` recovery — it does not conflict with SessionStart because
it explicitly excludes the SessionStart-only surface
(`brain/index/session-entry-contract.md`) and uses `session-resume.md` (the
SessionStart output) as a live re-read rather than duplicating the
bootstrap.

### 14.10 Proof run status ambiguity

**Symptom.** Task references a running proof run, but the current session
did not start it.

**Diagnose.**

```bash
brain-proof-status --limit 3
brain-proof-verify --run <RUN_ID>
```

If the run verifies `PASS`, close it by completing the associated task.
If it fails or is inconclusive, escalate to operator before finalizing.

### 14.11 Seat arbitration mismatch

**Symptom.** `brain/ops/.brain-active-seat.json` reports an `assistant`
value that does not match the running seat (e.g. a prior Codex session
was not released before a Claude session started).

**Diagnose first.**

```bash
cat brain/ops/.brain-active-seat.json
# Confirm: does the recorded assistant match the running session?
```

If the recorded assistant matches the running session, there is no
mismatch; stop. If it does not match, proceed to fix.

**Do not blindly re-run bootstrap.** The seat-entry bootstraps are **not**
safe to re-run in all conditions because they update seat claim, heartbeat,
and session registry state.

**Fix (narrow recovery -- stale-record case).** Use the direct seat-claim
tool, which only rewrites `.brain-active-seat.json` and has no remote
side effects:

```bash
brain-seat-claim --seat laptop --assistant <current-agent> --mode interactive
brain-heartbeat 2>/dev/null || true
```

`<current-agent>` is `claude-code`, `codex`, or `gemini`. This is the
only step needed when the mismatch is a stale record from a prior
session that ended without clean release.

**Fix (full recovery).** Re-run the full seat bootstrap for the current seat
when the stale-record fix above does not reconcile seat state.

**Never hand-edit `.brain-active-seat.json`.** Always go through a tool.
